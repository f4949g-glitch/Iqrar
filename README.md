# إقرار

منصة ويب (TypeScript + React + Vite + Tailwind، متصلة بـ Supabase) لخدمة **توثيق العقود**: إنشاء عقود متعددة الأطراف، إرسالها للتوثيق برابط توقيع فريد لكل طرف، تعبئة حقول ديناميكية موضوعة فوق مستند PDF، توليد نسخة PDF نهائية بعد اكتمال كل التواقيع، ومتابعة الحالة وسجل الأحداث.

## الإعداد والتشغيل المحلي

```bash
npm install
npm run dev          # خادم التطوير (عادة http://localhost:5173)
npm run typecheck    # التحقق من عدم وجود أخطاء TypeScript
npm test             # تشغيل الاختبارات
npm run build        # بناء نسخة الإنتاج
```

`.env` جاهز بقيم مشروع `iqrar` على Supabase (`chdvqkvfdxjfiitexcps`، منسوخة من `.env.example`، قيم publishable عامة وليست سرّية).

## بنية المشروع

```
src/
  app/            شل التطبيق: التوجيه، الـ Layout، الصفحة الرئيسية (لوحة الخدمات)
  features/
    auth/         تسجيل الدخول بالبريد الإلكتروني (حساب المدير الوحيد حاليًا) + تغيير كلمة المرور الإجباري
    contracts/    معالج إنشاء العقد (أطراف → رفع PDF → وضع الحقول → مراجعة وإرسال)، قائمة العقود
                  بثلاث تبويبات (جديدة/سابقة/تتطلب توثيقي)، صفحة تفاصيل العقد
    signing/      صفحة توقيع عامة بلا تسجيل دخول (/sign/:token) لأطراف العقد الخارجيين
  shared/ui/      عناصر واجهة عامة (Button, Field, SignaturePad, StatusPill)
  lib/supabase/   عميل Supabase + الأنواع المولَّدة من قاعدة البيانات
  lib/pdf/        إعداد pdf.js worker لعرض PDF داخل المتصفح (react-pdf)

supabase/
  migrations/     مخطط قاعدة البيانات كاملًا (SQL)، مُطبَّق فعليًا على المشروع
  functions/      Edge Functions:
                  get-signing-session   قراءة عامة لجلسة توقيع طرف عبر التوكن
                  submit-signature      اعتماد توقيع الطرف + توليد PDF النهائي عند اكتمال الكل
                  send-contract-notifications   إشعارات بريدية عند إرسال العقد للتوثيق
```

## قاعدة البيانات

- **المشروع**: `iqrar` (`chdvqkvfdxjfiitexcps`) على Supabase، مؤسسة `f4949g-glitch's Org`.
- **RLS مفعّلة على كل الجداول**. المستخدم المصادَق الوحيد حاليًا هو "مدير" (`profiles.role = admin`)؛ التسجيل العام غير مفعَّل في الواجهة (لا صفحة تسجيل)، وأي حساب جديد لأطراف العقود سيُنشأ لاحقًا عبر Edge Function بصلاحية service_role (مرحلة 2).
- **الجداول**: `profiles`، `contracts`، `contract_parties` (رابط توقيع فريد `token` لكل طرف)، `contract_fields` (حقول ديناميكية بموضع نسبة مئوية من الصفحة)، `contract_events` (سجل تدقيق)، `pricing_settings`.
- **تخزين الملفات**: bucket خاص باسم `contracts` (غير عام)؛ الوصول للمستند الأصلي والنهائي عبر روابط موقّعة مؤقتة، وأطراف التوقيع العامون يصلون عبر Edge Functions بصلاحية service_role فقط (لا وصول مباشر من anon لسياسات Storage).
- **الحساب الحالي**: `f4949g@gmail.com` مربوط بصف `profiles` بدور `admin` (كلمة مرور مؤقتة، يُطلب تغييرها عند أول دخول).

## نطاق ما أُنجز حتى الآن (المرحلة 1)

- إنشاء عقد: بيانات الأطراف (صفة + اسم + هوية/إقامة + بريد + جوال) مع تسعير فوري (سعر لكل طرف + حد أدنى للفاتورة)، رفع PDF، وضع حقول ديناميكية بالسحب فوق صفحات المستند (توقيع/نص/رقم/بريد/جوال/تاريخ/وقت/صورة/شعار/ختم/خانة اختيار/قائمة/نص متعدد الأسطر/ملف)، مراجعة، وإرسال للتوثيق.
- توقيع الأطراف عبر رابط فريد بلا تسجيل دخول، مع توليد نسخة PDF نهائية مدمجة بكل القيم المعبّأة فور اكتمال توقيع الجميع.
- إشعار بريدي (عبر Resend إن أُضيف `RESEND_API_KEY` كسرّ لدوال Supabase Edge؛ يعمل النظام بدونه لكن بدون إرسال فعلي — الرابط يبقى متاحًا للنسخ اليدوي من صفحة تفاصيل العقد).
- قائمة العقود بثلاث تبويبات، صفحة تفاصيل بسجل الأحداث ونسبة الإنجاز وتحميل النسخة النهائية.
